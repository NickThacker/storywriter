# Phase 01.1: Remove n8n - Research

**Researched:** 2026-03-01
**Domain:** Codebase cleanup — removing n8n webhook client, test endpoint, and security perimeter code from a Next.js + Supabase project
**Confidence:** HIGH

## Summary

Phase 01.1 is a surgical deletion and simplification pass. The decision to remove n8n is already made and documented in STATE.md: n8n's 30-second webhook timeout blocks all heavy AI tasks (outline: 60-120s, chapters: 60-300s), streaming cannot relay through webhooks, and all generation already routes directly through Next.js Route Handlers calling OpenRouter. n8n was never actually used for any AI work; it was a security perimeter stub that added complexity without value.

The scope is narrow and well-defined: three source files to delete/simplify, two env vars to remove from `.env.local`, and one already-staged deletion of `.env.local.example` to commit. The most important downstream concern is **Phase 2 Plan 02-04**, which was written expecting `src/lib/n8n/client.ts` to exist — the premise-prefill API route imports `triggerN8nWorkflow` and `isN8nConfigured` from that module. After removal, that route must call OpenRouter directly (for configured API keys) or return a mock (for unconfigured). This plan must update 02-04-PLAN.md accordingly.

The ROADMAP.md also has residual n8n references in Phase 4 ("n8n Wait node HITL") that should be scrubbed to prevent future confusion. Planning documents and summaries contain n8n references but those are historical records — they should be left alone unless they cause active confusion.

**Primary recommendation:** Delete the three n8n files, simplify the health route, clean env vars, update 02-04-PLAN.md's premise-prefill task to use direct OpenRouter instead of `triggerN8nWorkflow`, and scrub forward-looking n8n references from ROADMAP.md.

## Architecture Patterns

### Pattern 1: Direct File Deletion

**What:** Delete source files that have no callers outside themselves.

**Verification before deletion:**
```bash
# Confirm nothing else imports from n8n/client
grep -r "from.*n8n" src/
grep -r "n8n/client" src/
```

Expected output: only the three files in the inventory above. If any other file imports from n8n, it must be fixed before the deletion.

**Then delete:**
```bash
rm src/lib/n8n/client.ts
rm src/app/api/n8n/test/route.ts
rmdir src/app/api/n8n/
```

### Pattern 3: Update 02-04-PLAN.md Premise-Prefill Task

**What:** The premise-prefill route was written to call `triggerN8nWorkflow` when n8n is configured and return a mock otherwise. After removing n8n, both branches must use direct OpenRouter calls or the mock-only path.

**Decision:** Since premise prefill is a short inference call (< 10s), it should call OpenRouter directly from the Route Handler using the existing server-side API key retrieval pattern. The mock path (for users with no API key configured) stays.

**Updated premise-prefill route logic:**
```typescript
// src/app/api/generate/premise-prefill/route.ts
import { NextResponse } from 'next/server'
import { createClient } from '@/lib/supabase/server'

export async function POST(request: Request) {
  const supabase = await createClient()
  const { data: { user }, error } = await supabase.auth.getUser()
  if (error || !user) return NextResponse.json({ error: 'Unauthorized' }, { status: 401 })

  const body = await request.json()
  const { premise } = body
  if (!premise) return NextResponse.json({ error: 'Missing premise' }, { status: 400 })

  // Retrieve API key server-side
  const { data: settings } = await (supabase as any)
    .from('user_settings')
    .select('openrouter_api_key_ref')
    .eq('user_id', user.id)
    .single()

  // If no API key configured, return mock prefill for development/demo
  if (!settings?.openrouter_api_key_ref) {
    return NextResponse.json({
      genre: 'literary',
      themes: ['identity'],
      setting: 'urban-modern',
      tone: 'lyrical',
      characters: [{ role: 'protagonist', archetype: 'Protagonist' }],
    })
  }

  // Call OpenRouter directly for premise inference (< 10s, not streaming)
  // Retrieve actual key from Vault, then call OpenRouter
  // ... (standard OpenRouter fetch pattern, structured JSON output)
}
```

The 02-04-PLAN.md task must be updated to reflect this pattern — no `triggerN8nWorkflow` call, no `isN8nConfigured` check, direct OpenRouter call using the already-established BYOK pattern.

### Anti-Patterns to Avoid

- **Leaving orphaned import in health route:** TypeScript compile will fail if `src/lib/n8n/client.ts` is deleted while `src/app/api/health/route.ts` still imports from it. Must update health route before or in the same commit as deletion.
- **Deleting .env.local.example without recreating it:** The file is already staged as deleted in git. If left deleted, new developers have no template. Recreate without n8n vars.
- **Editing historical planning summaries:** Phase 01-03 exists and documents what was built. It should remain accurate for historical reference. Do not alter it.
- **Leaving n8n vars in .env.local:** The env file on disk still has the n8n block (lines 14-16). These are harmless but confusing — remove them.

## Don't Hand-Roll

| Problem | Don't Build | Use Instead | Why |
|---------|-------------|-------------|-----|
| Premise prefill AI call | Custom n8n-style orchestration layer | Direct `fetch` to OpenRouter API with `response_format: json_schema` | The call is < 10s, single-step — no orchestration needed |
| Security perimeter for AI calls | Custom secret-header middleware | Supabase auth (`getUser()`) already validates all server actions and route handlers | Auth is already established; n8n's webhook secret was solving a problem that Next.js auth already handles |

**Key insight:** The n8n security perimeter (shared secret header, IF node validation) was protecting n8n webhooks from unauthorized calls. With n8n gone, Next.js route handlers use Supabase `getUser()` which is the correct auth mechanism for this stack. No new security layer is needed.

## Common Pitfalls

### Pitfall 1: TypeScript Compile Failure from Orphaned Import

**What goes wrong:** Developer deletes `src/lib/n8n/client.ts` but forgets to update `src/app/api/health/route.ts`. TypeScript compilation fails with "Cannot find module '@/lib/n8n/client'". Build breaks.

**Why it happens:** Two files import from the n8n client — the test route (being deleted) and the health route (being kept). Easy to miss the health route.

**How to avoid:** Run `grep -r "n8n/client" src/` before deleting. Update or delete all importers in the same work unit.

**Warning signs:** `npx tsc --noEmit` fails immediately after deletion.

### Pitfall 2: Forgetting the .env.local File Has n8n Vars

**What goes wrong:** `.env.local.example` is cleaned up but `.env.local` (the actual secrets file, gitignored) still has `N8N_BASE_URL=` and `N8N_WEBHOOK_SECRET=`. Future developers may be confused about whether these are needed.

**Why it happens:** `.env.local` is gitignored so it's invisible to git diff and easy to forget.

**How to avoid:** Explicitly edit `.env.local` to remove lines 14-16 (the n8n comment + two vars) as part of this phase.

### Pitfall 3: 02-04-PLAN.md Becomes Invalid

**What goes wrong:** Phase 02-04-PLAN.md is executed as written and fails because it imports from the now-deleted `src/lib/n8n/client.ts`. The executor follows the plan, hits a missing module error, and must debug mid-execution.

**Why it happens:** 02-04-PLAN.md was written before the n8n removal decision. It explicitly references `triggerN8nWorkflow` and `isN8nConfigured`.

**How to avoid:** Update 02-04-PLAN.md's Task 2 during this phase to remove n8n references and describe the direct OpenRouter call pattern instead.

### Pitfall 4: Removing n8n Vars Breaks ROADMAP.md Todos in STATE.md

**What goes wrong:** STATE.md Pending Todos section still references "Set N8N_BASE_URL and N8N_WEBHOOK_SECRET in .env.local". These become invalid instructions after cleanup.

**Why it happens:** STATE.md pending todos are written during plan execution and not automatically updated when plans are superseded.

**How to avoid:** Update or remove the n8n-related pending todos from STATE.md as part of this phase:
- "Set N8N_BASE_URL and N8N_WEBHOOK_SECRET in .env.local to test n8n pipeline"
- "Create n8n test workflow (POST /webhook/test with IF node secret validation) to exercise /api/n8n/test"
